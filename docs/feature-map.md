# aivm Feature Map

> Generated by static analysis of the codebase. No speculative additions; every claim is grounded in source code.

---

## 1. High-Level System Overview

**aivm** is a macOS CLI tool that manages a secure, disposable [Colima](https://github.com/abiosoft/colima) virtual machine specifically for running AI coding agents (Claude Code or GitHub Copilot). A single command drives the full lifecycle: VM creation, toolchain bootstrapping, agent launch, idle suspension, and eventual teardown.

The runtime architecture at steady state:

```
HOST (macOS)
├── aivm CLI (Go binary)           — user-facing entry point for all operations
├── MCPJungle (Docker container)   — MCP gateway on 127.0.0.1:7593
└── Idle Monitor daemon            — background process managing VM suspension/deletion

VM (Colima "aivm" profile)
├── AI agent (Claude Code | Copilot)
├── Toolchain (Java, Maven, Node.js, Python, Go, …)
└── rtk (token-optimizer proxy)
```

The host dev directory (e.g. `~/dev`) is bind-mounted read-write into the VM at the same absolute path. Claude project history is also persisted on the host at `~/.aivm/.claude/projects/` so it survives VM deletions.

## 2. Feature Catalog

### 2.1 `aivm` (default command) — Start + Launch

| Attribute | Detail |
|---|---|
| **Entry point** | `aivm [directory]` — root Cobra command with no subcommand |
| **Source** | `internal/cli/root.go`, `internal/cli/start.go`, `internal/cli/launch.go` |
| **What it does** | Calls `DoStart` then `DoLaunch` in sequence. Starts the VM and MCPJungle if needed, runs bootstrap/sync, then opens an interactive AI agent session. |
| **Dependencies** | VM (Colima), MCPJungle, Plugin system, Agent provider, Session store, Idle monitor |
| **Configuration** | `vm.*`, `mcp.*`, `idle.*`, `agents.*`, `plugins.*` |
| **Complexity** | **High** — orchestrates DoStart (itself high complexity) and DoLaunch; includes transition routing, working-directory validation, session lifecycle, signal handling |

### 2.2 `aivm start` — VM & Service Start

|---|---|

### 2.8 `aivm rebuild-image` — Base Image Rebuild

| Attribute | Detail |
|---|---|
| **Entry point** | `aivm rebuild-image [--force]` subcommand |
| **Source** | `internal/cli/rebuild_image.go: DoRebuildImage()` |
| **What it does** | Rebuilds the base VM snapshot from scratch. | **Configuration** | `vm.*`, `plugins.*` |
| **Complexity** | **High** — two distinct rebuild paths; temp-VM lifecycle management; transition state serialization; base image metadata vs full snapshot distinction |

### 2.12 Base Image System

| Attribute | Detail |
|---|---|
| **Source** | `internal/vm/image.go` |
| **What it does** | Maintains `~/.aivm/base-image.json` (ID + snapshot name + timestamp). After a successful bootstrap, saves a Colima snapshot as the base image. ### 2.13 VM Transition / Dual-VM Migration

| Attribute | Detail |
|---|---|
| **Source** | `internal/vm/transition.go`, `internal/cli/launch.go` (startTransitionVM, rebuildCurrentVM), `internal/cli/rebuild_image.go` (doSoftRebuild) |
| **What it does** | When a base image rebuild is requested while sessions are active, a second VM is created (profile `<name>-next` or `<name>-rebuild`), bootstrapped, and recorded as the new target. A `vm-transition.json` state file routes future `aivm launch` calls to the new VM. The legacy VM continues serving existing sessions. | **Dependencies** | VMFactory, ImageManager, session store, idle monitor (legacy monitor path), TransitionState file |
| **Configuration** | `vm.*` (for new VM creation) |
| **Complexity** | **High** — manages two concurrent VM profiles; state-machine across files (transition JSON + session locks + image ref); two separate daemon types (idle monitor + legacy monitor); edge cases for interrupted transitions |

### 2.15 Bootstrap Engine

|---|---|
| **Source** | `internal/bootstrap/bootstrap.go` |

### 2.16 Plugin System

| Attribute | Detail |
|---|---|
| **Source** | `internal/plugin/plugin.go`, `internal/plugin/yaml_plugin.go`, `internal/plugin/registry.go`, `internal/plugin/dag.go`, `internal/plugin/executor.go`, `internal/plugin/defaults.yaml` |
| **What it does** | A plugin is a named unit with `check`, `install`, and `configure` shell scripts plus a `dependencies` list and optional `agents` filter. Scripts are Go templates rendered against per-plugin config. The registry stores plugins by name (upsert semantics). `Resolve()` expands a list of enabled plugins transitively via DFS then topologically sorts them with Kahn's algorithm. `Executor.Run()` iterates in order, calls Check, skips if installed, otherwise calls Install then Configure. |
| **Dependencies** | ColimaVM (script runner via `InstallEnv.VM`), Go `text/template` |
| **Configuration** | `plugins.enabled`, `plugins.config.<name>`, `plugins.define` (user overrides), `agents.config` (agent-specific plugin defs from `internal/agent/defaults.yaml`) |
| **Complexity** | **Medium** — DAG resolution, template rendering, three-phase lifecycle (check/install/configure), dual execution mode (in-VM via SSH vs. host-side bash) |

### 2.18 Integration System

| Attribute | Detail |
|---|---|
| **Source** | `internal/integration/integration.go`, `internal/integration/defaults.yaml` |
| **What it does** | Integrations are conditional configure scripts that run when a specific plugin is installed **and** a specific agent is active. Scripts support Go templates with `{{ .mcp_port }}` injection. Keys (`from:to` or `name`) are tracked in `bootstrap-state.json` so each integration runs exactly once. | **Dependencies** | ColimaVM (script runner), bootstrap state file |
| **Configuration** | `integrations` list in `aivm.yaml` (user-defined); built-in integrations embedded in `defaults.yaml` |
| **Complexity** | **Medium** — three matching dimensions (plugin, agent, already-ran); template rendering; idempotency bookkeeping; agent-only vs. plugin-gated distinction |

### 2.23 MCPJungle Service

| Attribute | Detail |
|---|---|
| **Source** | `internal/mcp/compose.go`, `internal/mcp/docker.go`, `internal/mcp/manager.go` |

## 3. Configuration System Deep Dive

### 3.1 YAML Structure Breakdown

```
aivm.yaml
├── vm
│   ├── cpus                      int      VM CPU count
│   ├── memory                    int      VM RAM in GiB
│   ├── disk                      int      VM disk in GiB
│   ├── type                      string   "vz" (Apple Silicon) | "qemu" (Intel)
│   ├── max_age_days              int      Age before prompting to recreate VM (0 = disabled)
│   ├── base_image_max_age_days   int      Age before prompting to rebuild base image (0 = disabled)
│   ├── dev_root                  string   Host dev directory (mounted at same path in VM)
│   └── profile                   string   Colima profile name (default: "aivm")
│
├── mcp
│   ├── port                      int      Host port for MCPJungle (default: 7593)
│   ├── data_dir                  string   Persistent SQLite data for MCPJungle
│   ├── image_tag                 string   Docker image tag for MCPJungle
│   └── server_mode               string   MCPJungle server mode ("development")
│
├── idle
│   ├── timeout                   duration VM suspend after N idle (default: 5m)
│   └── delete_timeout            duration VM delete after N suspended (default: 5m)
│
├── agents
│   ├── enabled                   string | []string  Active agent ("claude" | "copilot")
│   ├── config                    map[name]map[key]any  Per-agent options
│   │   └── copilot.launch_command  string  CLI command for Copilot (default: "gh copilot")
│   └── define                    map[name]map[key]any  (reserved, unused)
│
├── plugins
│   ├── enabled                   []string  List of plugin names to install
│   ├── config                    map[name]map[key]any  Per-plugin template vars
│   │   ├── java.version          string    JDK version (default: "25")
│   │   ├── java.distribution     string    JDK distribution (currently unused)
│   │   ├── nodejs.version        string    Node.js version (currently unused in script)
│   │   └── python.tool           string    Python installer (currently unused in script)
│   └── define                    map[name]PluginDef  Custom/override plugin definitions
│
├── integrations                  []IntegrationDef  User-defined integrations
│   ├── name                      string    Idempotency key (required when from is empty)
│   ├── from                      string    Plugin prerequisite (empty = agent-only)
│   ├── to                        string    Agent name
│   ├── when                      string    "installed" (only supported value; informational)
│   └── configure                 string    Shell script (Go template, {{ .mcp_port }} available)
│
└── debug                         bool      Enable debug logging (can also use --debug flag)
```

### 3.2 Agents System

Agents are **runtime providers** selected by `agents.enabled`. At most one agent is active at a time (first entry wins). Two providers are registered at startup:

- **`claude`** (`internal/providers/claude/provider.go`): requires the `claude` plugin; runs `claude --dangerously-skip-permissions --mcp-config ~/.claude/mcp-config.json`
- **`copilot`** (`internal/providers/copilot/provider.go`): requires the `copilot` plugin; runs `gh copilot` (or custom `launch_command`)

The agent plugin (`claude` or `copilot`) is **automatically appended** to the effective plugin list by `bootstrapEnabledPlugins()` — it does not need to appear in `plugins.enabled`.

`agents.config.<name>` is passed verbatim as `LaunchEnv.Config` to the provider at launch time. Currently only `copilot.launch_command` is consumed.

`agents.define` exists in the struct but is neither populated nor consumed anywhere in the codebase (reserved for future use).

### 3.3 Integrations System

The integration system is a **post-bootstrap configuration layer**. After plugins are installed, `runIntegrationsFromState()` is called with the full integration list (built-in defaults + user-defined). The executor filters to matching integrations and runs their configure scripts inside the VM.

**Execution conditions for a match:**
1. `integ.From` is in `InstalledPlugins` set (or `From` is empty → agent-only)
2. `integ.To` is in `ActiveAgents` slice
3. `integ.Key()` is not in `AlreadyRan` set

Template variables available in configure scripts: `{{ .mcp_port }}` (MCP port number).

**Idempotency:** Integration keys (`from:to` or `name`) are stored in `bootstrap-state.json` under `"integrations"`. Each key runs exactly once per VM lifecycle. Clearing bootstrap state (e.g. after base image restore) causes integrations to re-run.

**Merge order for the integration list:**
```
built-in (integration/defaults.yaml) → cfg.Integrations (aivm.yaml)
```
User-defined integrations are appended; they do not override built-ins by key.

### 3.4 Plugins System

Plugin definitions are loaded and merged in three layers:

```
1. plugin/defaults.yaml   (embedded in binary)
2. agent/defaults.yaml    (embedded; agent-specific plugins like "claude", "copilot")
   → merged field-by-field into layer 1 (MergePluginDef)
3. cfg.Plugins.Define     (from aivm.yaml plugins.define)
   → merged field-by-field into result (MergePluginDef)
```

`MergePluginDef` semantics: non-zero fields in the override win; `Defaults` maps are merged key-by-key.

**Effective plugin list construction (bootstrapEnabledPlugins):**
```
cfg.Plugins.Enabled + provider.RequiredPlugins() (de-duped, order preserved)
```

**DAG resolution (`Registry.Resolve`):**
1. Expand each enabled plugin's `dependencies` recursively (DFS)
2. Build dependency subgraph
3. Topological sort (Kahn's algorithm with alphabetical tie-breaking for determinism)

**Plugin execution (`Executor.Run`):**
- Per-plugin config = `PluginDef.Defaults` merged with `cfg.Plugins.Config[name]` (user values win)
- Config values are injected as Go template data into script strings
- Each script runs in the VM via `colima ssh` base64-encoded bash

### 3.5 Interactions Between Agents, Integrations, and Plugins

```
                          ┌─ plugins.enabled ──────┐
                          │  + provider.Required    │
                          ▼                         │
                  Plugin DAG Resolution             │
                          │                         │
                  Bootstrap execution               │
                  (install + configure)             │
                          │                         │
                   BootstrapState.Installed ─────────┤
                          │                         │
                          ▼                         │
              Integration matching                  │
              (InstalledPlugins × ActiveAgents)      │
                          │                         │
              Integration configure scripts         │
              → MCP config injection                │
              → rtk global wiring                   │
                          │                         │
               BootstrapState.Integrations ─────────┘
```

Key design invariant: **plugins know nothing about agents; agents know nothing about plugins**. The integration layer is the sole bridge, and it runs after all plugins complete.

## 4. Complexity Hotspots

### 4.1 Bootstrap State Machine (`internal/cli/agent_vm.go`)

**Coupling level: High**

`agent_vm.go` is the central orchestration hub and has the most interlocking responsibilities in the codebase:
- Maintains host-side bootstrap state (`BootstrapState` read/write/clear)
- Owns `syncBootstrap` — the main entry point that branches into four distinct paths (no-op, new plugins, agent mismatch, full bootstrap)
- Owns `fullBootstrap`, `rebuildBootstrap`, `recreateVMForConfiguredAgent`
- Triggers integration execution
- References the bootstrap engine, plugin registry, agent registry, session store, VM, VM factory, and image manager

This file is the highest-complexity module and the most difficult to modify safely.

### 4.2 `DoStart` (`internal/cli/start.go`)

**Coupling level: High**

`DoStart` orchestrates: MCPJungle start → VM age check → VM create/resume → readiness wait → base image restore attempt → bootstrap selection → idle monitor start. Each of these branches into sub-paths. It also writes the `vm-created-at` file, manages `.claude/projects` directory creation, and clears `vm-stopped-at` on resume.

### 4.3 `DoLaunch` + `checkBaseImageAge` (`internal/cli/launch.go`)

**Coupling level: High**

`DoLaunch` has a hidden side effect: before launching the agent it calls `checkBaseImageAge`, which can interactively trigger either a full VM rebuild or a two-VM transition, fundamentally mutating `app.VM`. This makes `DoLaunch` an unexpected entry point for large structural changes.

### 4.4 VM Transition Logic (split across three files)

**Coupling level: High**

The transition/dual-VM feature spans:
- `vm/transition.go` — state serialization
- `cli/launch.go` — `startTransitionVM`, `rebuildCurrentVM`, `checkBaseImageAge`
- `cli/rebuild_image.go` — `doSoftRebuild`
- `monitor/idle.go` — `RunLegacyMonitor`, `EnsureLegacyMonitorRunning`
- `cli/root.go` — `legacyMonitorCmd`

No single file owns the full transition lifecycle, making it hard to reason about or modify.

### 4.5 Bootstrap State as Dual-Purpose Tracker

**Potential redundancy**

`BootstrapState` in `bootstrap-state.json` tracks both installed plugins **and** executed integrations in a single JSON file. These are conceptually separate concerns with different invalidation rules (plugins survive resume; integrations are re-evaluated after base image restore). Combined storage means a full state clear (e.g. for base image restore) also resets integration tracking, causing integrations to re-run.

### 4.6 `syncBootstrap` Complexity

`syncBootstrap` handles five distinct cases in sequence:
1. No state file or stale schema version → `fullBootstrap(force=false)`
2. Required agent plugins missing, but another agent's plugins present → `resolveAgentMismatch`
3. Required agent plugins missing, no other agent detected → fall through to new-plugins path
4. No new plugins → skip bootstrap, run integrations
5. New plugins detected → install new plugins only, update state, run integrations

The interleaving of agent-mismatch detection and plugin-diff logic in one function is a significant readability and testability burden.

### 4.7 Interactive Mode / Terminal Detection

**Coupling level: Medium**

Interactive prompts are scattered across `start.go` (`shouldRecreateVM`), `launch.go` (`checkBaseImageAge`), `agent_vm.go` (`resolveAgentMismatch`), and `rebuild_image.go`. Each calls `interactive(app)` independently. The injectable `app.IsTerminal` / `app.Stdin` fields allow testing but require manual wiring at every callsite.

### 4.8 Plugin Config — Partially Unused Fields

`plugins.config.nodejs.version` and `plugins.config.python.tool` are accepted by the configuration system and exposed as template variables, but the corresponding plugin install scripts in `defaults.yaml` do not use `{{ .version }}` or `{{ .tool }}` — they hardcode `setup_lts.x` and the `uv` installer URL. The `java.distribution` field is similarly accepted but unused in the install script.

### 4.9 `agents.define` Dead Code

`AgentsConfig.Define` (`map[string]map[string]any`) is present in the struct and unmarshalled from config, but is never read or acted upon anywhere in the codebase.

### 4.10 MCP `server_mode` Field

`mcp.server_mode` is passed to the MCPJungle container as `SERVER_MODE=development` but has no documented effect on aivm's own behaviour. Its meaning is internal to MCPJungle.

### 4.11 `config/colima.yaml` — Dead Configuration File

`config/colima.yaml` is a reference/template file. Its own comment states it is "NOT directly consumed by Colima as a config file." It does not appear to be read anywhere in the Go codebase. It is a documentation artifact, not a live configuration file.
